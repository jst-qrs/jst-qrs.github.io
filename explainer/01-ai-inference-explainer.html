The Building Blocks

Before the pipeline makes sense, these core concepts need to click: what a GPU actually is, why matrix multiplication dominates everything, what precision formats trade off, and where the model physically lives while it runs.

What Is a GPU, Really?
Built for parallel arithmetic, not sequential logic

A CPU has a handful of powerful cores (8–64) optimised for doing many different tasks quickly, one after another, with complex branching logic. A GPU has thousands of small, simple cores optimised for doing the same arithmetic operation on massive amounts of data simultaneously. A CPU is a small team of experts; a GPU is a stadium of workers all multiplying numbers in lockstep.

Neural networks are, at their core, enormous sequences of multiply-and-add operations on grids of numbers. That happens to be exactly the shape of problem a GPU was originally built to solve — rendering pixels, which are also just grids of numbers being transformed in parallel. AI didn't reshape GPUs; it discovered they already fit.

CUDA Cores
General-purpose parallel arithmetic units

The "default" GPU core. Each one does one scalar operation (add, multiply) per clock cycle — FP32 or INT32 math. An H100 has 16,896 CUDA cores. Good for graphics, general parallel compute, and any workload that isn't specifically matrix multiplication.

Scalar opsFP32 / INT32General purpose
Tensor Cores
Purpose-built matrix multiply-accumulate units

Introduced by NVIDIA in 2017 specifically for AI. A single tensor core multiplies two small matrices (e.g. 4×4) and accumulates the result in one operation — work that would take a CUDA core dozens of cycles. An H100 has 528 tensor cores delivering most of its AI throughput, even though it has far fewer of them than CUDA cores.

Matrix opsFP16/BF16/FP8/INT8AI-specific
The practical effect: nearly all LLM inference compute runs on tensor cores, not CUDA cores. CUDA cores handle the surrounding glue logic — indexing, control flow, activation functions — while tensor cores chew through the matmuls that dominate runtime.
Why Does Everything Reduce to Matrix Multiplication?
The one operation that defines AI hardware design

Every layer in a transformer is, mechanically, a series of matrix multiplications. The embedding lookup, the Q/K/V projections in attention, the attention score computation itself, the feed-forward network's two linear transforms, the final vocabulary projection — all matmuls. A "70 billion parameter model" literally means 70 billion numbers arranged into weight matrices that get multiplied against your input vectors, layer after layer.

This matters because matrix multiplication has a very specific computational signature: it's highly parallel (every output element is computed independently), regular (same operation repeated millions of times, easy to pipeline), and memory-hungry (every element of both matrices typically needs to be read). That signature is exactly what GPUs, TPUs, and every AI accelerator are designed around — which is why "more tensor core throughput" and "more memory bandwidth" are the two numbers vendors compete on above all else.

FLOP/s — The Compute Speedometer
Floating Point Operations per Second

A FLOP is one add or multiply on a floating point number. FLOP/s measures how many of these a chip can do per second — the standard unit for comparing raw AI compute power. A matmul between an M×K matrix and a K×N matrix takes roughly 2×M×K×N FLOPs (multiply + add for each element). For a 70B model processing one token, a single forward pass is on the order of 140 billion FLOPs.

But raw FLOP/s is a ceiling, rarely the reality. Actual achieved FLOP/s ("MFU" — Model FLOPs Utilisation) depends heavily on whether the chip can be fed data fast enough — which is where memory bandwidth comes back in. A chip with huge FLOP/s and slow memory will sit half-idle waiting for data, especially during decode.

Numeric Precision Formats
How numbers are represented — the core size/speed/accuracy lever

Every weight and activation in a neural network is a floating point number, stored using some number of bits split between sign, exponent (range), and mantissa (precision). Fewer bits means less memory, less data to move, and faster math — at the cost of numeric precision.

FORMATBITSSTRUCTUREPRIMARY USEWHY
FP32 32 1 sign · 8 exp · 23 mantissa Training (master weights), scientific compute Highest precision — needed to accumulate tiny gradient updates without losing signal over millions of steps
FP16 16 1 sign · 5 exp · 10 mantissa Inference, mixed-precision training Half the memory of FP32, fast on tensor cores — but narrow exponent range risks overflow/underflow in training
BF16 16 1 sign · 8 exp · 7 mantissa Training standard today, inference Same exponent range as FP32 (no overflow issues) with less mantissa precision — the training sweet spot
FP8 8 1 sign · 4 exp · 3 mantissa (E4M3 variant) Inference on H100+/B200, some training Quarter the memory of FP32. NVIDIA's Transformer Engine auto-casts to FP8 for matmuls where precision loss is tolerable
INT8 / INT4 8 / 4 Fixed-point integer Post-training quantised inference Smallest footprint, fastest — weights are quantised after training using calibration data to minimise quality loss
🏋️ Training Precision
Why training needs more numeric headroom

Training repeatedly computes gradients and nudges billions of weights by tiny amounts, accumulated over millions of steps. If precision is too low, small updates get rounded to zero and the model stops learning — a problem called "underflow." This is why training typically runs in BF16 or FP16 for the forward/backward pass, while keeping an FP32 master copy of the weights that absorbs the small updates accurately, then casts back down for the next step. This pattern is called mixed-precision training.

🚀 Inference Precision
Why inference can go much lower

Inference only runs the forward pass once per token — no gradients, no accumulation of tiny updates over time. The model's knowledge is already baked into the weights; a small rounding error in any single forward pass barely moves the output. This tolerance is what makes aggressive quantisation viable: FP8, INT8, even INT4 are common for serving models, trading a small, often imperceptible quality dip for 2–4× less memory and meaningfully faster, cheaper serving.

Where Does the Model Actually Live?
In GPU memory (HBM) — and it has to stay there

When a model is "loaded" for serving, its weights — every one of those 70 billion-plus numbers — are copied from disk into the GPU's onboard memory, called HBM (High Bandwidth Memory). They stay resident there for as long as the model is serving traffic. This is non-negotiable, not a convenience:

~1.5 GB/s
Typical PCIe/disk transfer to load weights once at startup
~3.35 TB/s
H100 HBM bandwidth — what the GPU reads weights at, every single forward pass
2,200×
Roughly how much faster HBM is than re-reading from disk every step

If weights had to be streamed from disk or even system RAM on every token, inference would be unusably slow — disk and even CPU RAM are orders of magnitude too slow to feed a GPU's compute units in real time. So the entire model must fit inside HBM simultaneously. This single constraint explains a huge amount of inference engineering: why quantisation matters (smaller weights fit on fewer GPUs), why models get sharded across multiple GPUs via NVLink (when one GPU's HBM isn't enough), and why HBM capacity is one of the first specs anyone checks on a new chip.

HBM (High Bandwidth Memory)
The GPU's "main" memory

3D-stacked DRAM chips mounted directly on the GPU package, connected via a very wide bus. Large capacity (80–192GB per chip) but, relative to on-chip memory, still "far" from the compute cores — every access has real latency. This is where the full model weights and the KV cache live.

80–192 GB~3–8 TB/sModel + KV cache
SRAM (On-Chip Cache)
Tiny, blazing-fast scratch space

Static RAM sitting directly on the compute die itself — the L1/L2 cache and shared memory tensor cores read from on every single operation. Minuscule capacity (tens of MB) but enormously faster than HBM, because there's essentially no distance for signals to travel. Techniques like Flash Attention exist specifically to keep intermediate computation inside SRAM and avoid expensive trips out to HBM.

~50 MB (H100)~20× HBM speedWorking scratchpad
Groq's LPU takes this to an extreme: instead of small SRAM + large HBM, it uses a huge 230MB of SRAM per chip and no HBM at all, holding model weights entirely on-chip across many chips. That's why it achieves such high single-stream token speeds — it sidesteps the HBM bottleneck entirely, at the cost of needing far more chips to hold a given model.
The Memory Bottleneck, in One Sentence

During decode, the GPU must read the entire model's weights from HBM just to generate a single token — so however fast HBM can deliver those bytes (its bandwidth, in TB/s) directly caps how many tokens per second the GPU can produce, regardless of how much raw compute (FLOP/s) it has sitting idle.

KV Cache — Quick Primer
Full interactive breakdown in the KV Cache tab →

Beyond the static model weights, HBM also holds the KV cache — the saved Key and Value vectors from attention for every token generated so far in a conversation. Without it, generating token N would require recomputing attention over all N-1 previous tokens from scratch, every single time. The KV cache trades memory for compute: it grows with every token generated and every concurrent user served, and at long context lengths it can outgrow the model weights themselves.

Advanced KV Cache Techniques
TECHNIQUE
Prefix Caching
System prompts and common prefixes are cached server-side across requests. If 1,000 users all start with the same system prompt, compute it once. Anthropic, OpenAI, and Google all offer this — often called "prompt caching." Can reduce TTFT by 80%+ for repeated prefixes.
TECHNIQUE
PagedAttention (vLLM)
KV cache stored in non-contiguous "pages" (like OS virtual memory). Eliminates memory fragmentation — previously 60–80% of GPU memory was wasted on internal fragmentation. vLLM's key innovation. Now standard in most serving systems.
TECHNIQUE
Speculative Decoding
A small "draft" model generates N tokens quickly, then the large model verifies them all in one parallel pass. If verified, all N tokens are accepted. Can achieve 2–3× throughput improvement with no quality loss. Google uses this for Gemini; called "Medusa" in some implementations.
TECHNIQUE
KV Quantisation
Compressing KV cache values from FP16 to INT8 or INT4. The KV cache can dwarf model weights at long contexts — quantising it halves or quarters memory, allowing more concurrent users or longer contexts per GPU.

Why Naive Batching Fails → Continuous Batching
PROBLEM
Static Batching
Old approach: batch N requests, run until ALL are done, then start the next batch. Problem: if user A needs 10 tokens and user B needs 1000 tokens, the GPU sits mostly idle waiting for B. GPU utilisation typically 30–40%.
INEFFICIENT
SOLUTION
Continuous Batching
As soon as any request finishes, a new request immediately fills that slot. The batch composition changes every iteration. Pioneered by vLLM. Achieves 23× higher throughput vs static batching in benchmarks. Now universal in production serving.
INDUSTRY STANDARD
ADVANCED
Chunked Prefill
Long prompts monopolise the GPU during prefill, starving decode requests. Chunked prefill splits large prompts into chunks interleaved with decode steps. Balances latency across users — no single long prompt can block everyone else.
PRODUCTION 2024
SCALING
Disaggregated Prefill/Decode
Run prefill (compute-bound) on dedicated GPU clusters, send KV cache over fast interconnect to decode (memory-bound) clusters. Each cluster can be independently scaled and optimised. Used at hyperscaler scale — Google, ByteDance, and emerging in open-source (Mooncake).

Where Inference Gets Stuck

The two phases of inference — prefill and decode — have fundamentally different bottlenecks. Understanding this is why hardware like Groq's LPU exists.

⚡ Prefill Phase
Processing all prompt tokens in parallel
BOTTLENECK: COMPUTE (FLOP/s)
Compute utilisation~85%
Memory bandwidth utilisation~40%

During prefill, many tokens are processed simultaneously — a large matrix × matrix operation that saturates CUDA cores. Memory bandwidth is comparatively underused. More FLOP/s = faster TTFT.

🔁 Decode Phase
Generating tokens one at a time
BOTTLENECK: MEMORY BANDWIDTH (TB/s)
Compute utilisation~8%
Memory bandwidth utilisation~92%

Each decode step processes just ONE token — a matrix × vector operation. The massive CUDA core array sits nearly idle while all the time is spent loading the model's weights (100s of GB) from HBM. More TB/s = more tokens/sec.

Model Parallelism — Sharding Across GPUs
STRATEGY
Tensor Parallelism
Weight matrices are split across GPUs column-by-column. Each GPU does a partial matmul, then an all-reduce combines results. Splits within a layer. Requires high-bandwidth interconnect (NVLink) — doesn't work well across nodes.
STRATEGY
Pipeline Parallelism
Different GPUs own different layers. GPU 1 handles layers 1–20, GPU 2 handles layers 21–40, etc. Minimal communication — only activations pass between GPUs. Works across nodes. Tradeoff: "pipeline bubbles" where GPUs wait for the previous stage.
STRATEGY
Expert Parallelism (MoE)
Mixture-of-Experts models (like Mixtral, GPT-4 rumoured) have many "expert" FFN layers, but only a few activate per token. Different GPUs own different experts. Tokens are routed to the right GPU. Enables huge parameter counts with manageable compute per token.

Inference Glossary

Every term you'll encounter in the inference stack, plain English.

TTFT
Time to First Token. Latency from prompt submission to receiving the first output token. Dominated by prefill compute time.
Tokens/sec
Output throughput during decode. Limited by memory bandwidth (how fast model weights load from HBM), not FLOP/s.
KV Cache
Stored Key and Value vectors from attention for all previous tokens. Avoids quadratic recomputation in the decode loop.
HBM
High Bandwidth Memory. 3D-stacked DRAM sitting on the GPU package. H100 has 80GB HBM3 at 3.35 TB/s. The main bottleneck for decode throughput.
Prefill
Processing the entire prompt in one parallel pass. Compute-bound. Produces the KV cache and the first output token.
Decode
Autoregressive token-by-token generation. Memory-bandwidth-bound. Each step loads all model weights to produce one token.
BPE
Byte-Pair Encoding. The most common tokenisation algorithm — merges frequent character pairs iteratively to create a subword vocabulary.
Quantisation
Reducing weight precision (FP32→INT8→INT4) to shrink model size and increase memory bandwidth efficiency. Trade quality for speed/size.
Tensor Parallelism
Splitting weight matrices across multiple GPUs so each does a partial matmul. Requires fast NVLink interconnect.
Continuous Batching
Serving strategy where finished requests are immediately replaced with new ones in the batch, keeping GPU fully utilised. Pioneered by vLLM.
PagedAttention
Storing KV cache in non-contiguous memory pages (like OS virtual memory), eliminating fragmentation waste. vLLM's core innovation.
Speculative Decoding
A small model drafts N tokens quickly; the large model verifies them all in one parallel pass. 2–3× throughput with identical output quality.
MoE
Mixture of Experts. Architecture where only a subset of FFN "experts" activate per token. Enables large parameter counts with less compute per token.
Flash Attention
Reorders attention computation to minimise HBM reads/writes by tiling operations to fit in SRAM. 3–5× faster and reduces memory from O(N²) to O(N).
vLLM
Open-source LLM serving engine by UC Berkeley. Introduced PagedAttention and continuous batching. Now the most widely deployed open inference server.
NVLink
NVIDIA's proprietary GPU-to-GPU interconnect. H100: 900 GB/s bidirectional. B200: 1.8 TB/s. Enables tensor parallelism within a node.
Logits
Raw unnormalised scores output by the final model layer — one per vocabulary token. Softmax converts these to probabilities for sampling.
Temperature
Sampling parameter that scales logits before softmax. High temperature = more random/creative. Low temperature = more deterministic/focused. 0 = greedy.
