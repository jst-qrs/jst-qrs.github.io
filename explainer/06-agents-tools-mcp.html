Agents, Tools & MCP

How a model breaks out of the context window, takes actions in the world, and why MCP exists as a standardisation layer.

What makes something an "agent"?
A plain LLM call is one-shot: prompt in, tokens out, done. An agent is when you put the model in a loop — it can decide to take an action (call a tool, search, run code), observe the result, and then continue reasoning. The model drives the loop. It decides when to use tools, which tool, with what inputs, and when it has enough information to stop and give a final answer.

The key insight: the context window IS the agent's working memory
Every tool call and its result gets appended to the context window as new tokens. The model doesn't have a separate "state" — everything it knows about what has happened lives in the growing token sequence. This is why long agentic tasks consume enormous context: 10 tool calls × average 500 tokens each = 5,000 tokens of "memory" just from the actions taken so far.
What is a tool, exactly?
A tool is a function the model is allowed to call. Before inference, the API sends the model a list of available tools — their names, descriptions, and parameter schemas. When the model decides to use one, it emits a structured JSON object (a tool call) instead of plain text. The host application intercepts that, runs the real function, and feeds the result back as a new message. The model never directly runs code or hits the internet — it asks, and the application does.
// How the model "sees" a tool — it's just a JSON schema in its context
{
  "name": "search_flights",
  "description": "Search for available flights between two cities on a date",
  "parameters": {
    "origin": { "type": "string" },
    "destination": { "type": "string" },
    "date": { "type": "string", "format": "date" }
  }
}
The description field is critical — the model reads it to decide when to use this tool. A well-written description is as important as the function itself.
Tool categories
🔍
Search / retrieval
Web search, vector DB lookup, SQL query. Lets the model access information beyond its training cutoff or internal knowledge.
web_search(query: str)
💻
Code execution
Run Python, bash, JS in a sandbox. Model writes code, executes it, reads the output — iterating until the task is done.
run_code(lang, code: str)
📁
File system
Read, write, list, delete files. Lets agents work with documents, configs, and data files as part of a longer task.
read_file(path: str)
🌐
HTTP / APIs
Call any REST endpoint. Book a flight, send an email, post to Slack, query a database — anything with an API.
http_request(url, method)
🖥️
Computer use
Take screenshots, click, type, navigate browsers. The model operates a real desktop UI like a human would.
screenshot() / click(x,y)
🤖
Sub-agents
Spawn another agent as a tool call. Multi-agent systems: an orchestrator delegates subtasks to specialist agents.
delegate(agent_id, task)

The context window fills up as the agent works
Each turn of the loop appends more tokens. The context window is the agent's only working memory — there is no hidden state. When the context fills up, the agent can't fit more history and must either summarise, truncate, or use external memory tools.
Memory types available to agents
📋 In-context (working memory)
Everything in the current token window. Immediate, zero-latency, but finite. Lost when the conversation ends.

The problem MCP solves
Before MCP, every tool integration was custom code: a different SDK, auth pattern, data format, and transport per service. If you wanted an agent to talk to GitHub, Slack, a database, and a browser — you wrote four completely different integrations. Multiply that by hundreds of AI apps and hundreds of services: an M×N explosion of bespoke connectors. MCP (Model Context Protocol, by Anthropic) is the standardisation layer that collapses that to M+N.

MCP architecture
MCP's three primitives
🔧 Tools
Functions the model can call that take action or fetch data. Equivalent to what we've been calling "tools" — but now standardised. The server advertises them; any MCP-compatible host can use them.
📄 Resources
Read-only data sources the model can read — like files, database records, API responses. The model requests them like URIs. Resources go into context without running as code.
💬 Prompts
Reusable prompt templates the server exposes. The host can insert them as structured messages — e.g. a "code review" prompt template that always follows the same format.
The complete picture — from user prompt to GPU to tool and back
This is the full stack. A user message triggers the agent loop. The context window assembles everything (system prompt, history, RAG, tools list). That token sequence hits the GPU — prefill, KV cache, decode. The model emits a tool_use block. The host routes it via MCP to an external server. The result comes back as tokens, appended to context. The loop repeats. Each iteration burns GPU compute and grows the KV cache.
Cost of every agent loop iteration
Each loop turn is a full GPU prefill+decode cycle. With a growing context, prefill gets more expensive every turn — you're processing all previous tokens again (though KV cache means only the new tokens actually run full compute; the rest are cache hits). Agentic tasks can cost 10–100× more tokens than a simple Q&A.
The model has no idea it's "in a loop"
From the GPU's perspective, every turn is just another inference call with a longer context. The "agent loop" is entirely orchestrated by the host application — it decides when to stop, when to pass tool results back, and whether to call the model again. The LLM just sees tokens in, produces tokens out.
